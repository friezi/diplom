\subsection{Das Verteilungsschema}
Ein Problem bei Dienstapplikationen im Internet ist eine starke Serverbelastung zu gewissen Stoßzeiten.
Die Masse der Anfragen an einen Server kann dazu führen, dass dieser der zeitgerechten Beantwortung nicht mehr nachkommen kann.
Im schlimmsten Fall
können einige Anfragen gar nicht beantwortet werden, da die Masse der Anfragen die zur Verfügung gestellte Speicherkapazität des Servers
übersteigt. Ein Server unterhält im Regelfall eine Warteschlange (Queue) in der diejenigen Anfragen vorgehalten werden, welche aufgrund
anderer in Abfertigung befindlicher Anfragen momentan nicht bearbeitet werden können. Wird die Kapazitätsgrenze dieser Warteschlange erreicht,
können weitere überschüssigen Anfragen nicht bearbeitet werden, sie werden verworfen.
Beim bestehenden Konzept zur Verteilung der RSS-Feeds kann es
vorkommen, dass Subscriber, deren Anfragen sich sehr weit hinten in der Warteschlange befinden, erst sehr spät bzw. zu spät die gewünschten
Informationen erhalten. Man denke z.B. an aktuelle Börsennachrichten, welche im Sekundenbereich aktualisiert werden. Hier kann eine Nachricht
schon als veraltet gelten, erreicht sie den Interessenten einige Sekunden zu spät. Subscriber, deren Anfragen den Server bei bereits
ausgeschöpfter Kapazität der Warteschlange erreichen, kommen gar nicht an die gewünschte Information.

Unser Ziel ist es, ein Verteilungsschema zu konzipieren, bei dem jeder Subscriber die gewünschte Information in angemessener bzw.
gewünschter Zeit erhält. Je früher ein Subscriber die neueste Information, die beim Server vorliegt, erhält, desto größer ist ihr Aktualitätsgrad.
Führen wir uns kurz vor Augen, welche Faktoren den Aktualitätsgrad beeinflussen können. Denkbar sind drei Situationen, die den Aktualitätsgrad
negativ beeinflussen:
\begin{enumerate}
  \item Anfrage des Subscribers an den Server kommt zu spät \label{enum:anf_zsp}
  \item Antwort des Servers erreicht den Subscriber zu spät \label{enum:ant_zsp}
  \item Subscriber erhält gar keine Antwort vom Server \label{enum:k_ant}
\end{enumerate}

Zu \ref{enum:anf_zsp}.: da wir es bei dem bestehenden Pull-Ansatz mit aktivem Polling der Subscriber zu tun haben, bewirkt eine höhere
Polling-Rate eine höhere Chance, dass Anfragen den Server mit geringer Verzögerung erreichen (bezogen auf neue vorliegende Informationen).
Eine höhere Polling-rate bedeutet mehr Anfragen, die den Server pro Zeiteinheit erreichen.\\

Zu \ref{enum:ant_zsp}.: zu spät erhaltene Antworten können darauf zurückzuführen sein (abgesehen von langen
Inter"-net"--Über"-tra"-gungs"-zeiten),
dass der Server mit der Beantwortung nicht nachkommt, seine Queue also zu voll ist. Abhilfe schafft hier eine Drosselung der Anfragen.\\

Zu \ref{enum:k_ant}.: um trotzdem an die gewünschten Informationen zu gelangen, muss dafür gesorgt werden, dass der Server nicht die einzige
Quelle ist, von der jene Informationen bezogen werden können.\\

Punkte \ref{enum:anf_zsp}. und \ref{enum:ant_zsp}. widersprechen sich zunächst. Eine Erhöhung der Anzahl der Anfragen kann also die Erfüllung
von \ref{enum:ant_zsp}. mit sich führen.
Zudem kann dies sogar die Erfüllung von \ref{enum:k_ant}. bewirken: bei einigen Servern ist es vorgesehen, die
Anfragen von Subscribern, welche in zu geringen zeitlichen Abständen auf den Server treffen, zu blocken. Wir suchen also nach einer Lösung, bei
der sich die drei Punkte nicht gegenseitig negativ beeinflussen, bzw. sich die Waage halten.
